## 筆者について
- 実務で新規サービスの企画・立ち上げに関わった経験はありません。
- AIの流行前から実務での開発を続けていますが、依然ジュニアレベルです。
- Webアプリケーション開発のみを前提にしています。

## 前提
以下はChatGPTとCodexを併用する想定で記載しています。
例えばClaudeとClaude Codeの組み合わせでは勝手が異なる可能性があります。

### ChatGPT と Codex の使い分けについて
ChatGPTはふわっとした指示からいい感じの成果物を作成するのが得意ですが、Codexはかっちり指示したことだけを愚直に作業する印象です。
以下のように使い分けるといいかもしれません。

- ChatGPT：ドキュメントの叩き台を作成
- Codex：実際のディレクトリ構成や既存の内容を踏まえた加筆・修正

（※なお筆者がChatGPTしか登録していないのはケチだからです）

## 企画書の作成

Notionでプロダクト用のワークスペースを作成し、その中に企画書ページを追加。
以下のフォーマットでプロダクトの構想をまとめていきました。

```md
## タイトル
サービス名。

### 由来
サービス名が造語ならその由来。

### キャッチコピー

### 解決する課題
どんな悩みを持つユーザーが利用するのかを箇条書きする。

### 概要
おおむねどんなことをするサービスかを箇条書きする。

##　ストーリー
ユーザーがサービスをどのように使うかを時系列順に書く。

1. ユーザーの課題発生
2. サービスへ登録orログイン
3. （以下、ユーザーの行動）

##　機能
思いつくまま機能を箇条書きする。

##　優先度
機能を優先度順で箇条書きする。（スケジュール作成時の参考）
```

書き終わったら、ChatGPTにレビューさせます。
AIはこちらに忖度する可能性があるため、厳しめにレビューするようプロンプトで指示しました。

```
以下の企画書を査読してください。
忌憚のない意見で、私に寄り添おうとしすぎずにはっきりといいところ・悪いところを指摘してください。
```

指摘が納得できるものであれば、その都度書き直して再度レビュー依頼。
最終的に指摘がなくなるまでこれを繰り返しました。

##　エージェントのインストール
私の場合はChatGPT Plusに登録しているので、Codexをインストールしてアカウントへログインしました。
完全無制限ではないものの定額でGPT-5.1を使い倒すことができます。

##　ドキュメント作成
以下を作成しました。

- README.md
- AGENTS.md

`AGENTS.md`にはCodexが作業する上での指示を書きます。
WebのChatGPTと壁打ちしつつ、以下のようなフォーマットで仕上げていきました。

（使いながらガンガン編集しているため、あくまで参考例として）

```md
# Repository Guidelines

## エージェント行動ルール（必ず遵守）
- このファイルに記載されたすべての規約に厳密に従うこと。
- 疑問点がある場合は必ず私へ質問してから作業すること。

### コード編集時
- Biome の指摘（lint・format・organizeImports を含む）は **全件必ず修正**すること。
- 生成・編集したコードは **Biome に完全準拠**させること。
- 「UI 実装方針」「インポート規約」「ディレクトリ構成」「コーディング規約」「アーキテクチャ方針」に従うこと。

### Git 操作
- コミットは「コミット方針」に厳密に従うこと。
- コミットは小さく、プッシュ前に必ず rebase -i でスカッシュすること。
- スカッシュ後のメッセージは 1 行で内容を明確に書くこと（プレフィックス不要）。

### レビュー
- 忖度せず、正確で客観的な指摘を行うこと。
- ただし、敬語および柔らかい言い回しを用いてストレスを与えないこと。

## UI 実装方針

## ディレクトリ構成

## ドキュメント構成

## コーディング規約

## コミット方針

## テスト方針

## セキュリティ・設定
```

### 要件定義書の作成

ルートに`/docs`ディレクトリを作成。その中に企画書と要件定義書を追加しました。

- 企画書：`docs/proposal.md`
- 要件定義書：`docs/requirements.md`

`requirements.md`には機能の要件をひとつずつまとめていきます。

```md
# 要件定義書

## 目次（見出しへのリンクを書く）
- [機能A](#機能A)
- [機能B](#機能B)
- [機能C](#機能C)

## 機能A

## 機能B

## 機能C
```

このフォーマットでは各見出し（機能）へのページ内リンクで目次を作っていますが、機能ごとにマークダウンファイルを分けてもいいと思います。

### システム設計
企画と要件に合致する技術スタックを選定します。

- アプリケーションのフレームワーク
- データベース
- ストレージ（画像など）
- サーバー

etc…

また、各コンポーネントがどのように連携するかの設計を行います。どうやって？

### READMEを記載
ここまで作成したドキュメントと技術スタックをもとに、READMEを作成します。
READMEからは企画書・要件定義書へリンクを張ります。

## ローカル環境の立ち上げ
選定したフレームワーク等でアプリケーションを立ち上げます。
私の場合はこだわりがなければNext.jsを使用しています。

データベースやストレージなど、アプリケーションと連携するコンポーネントはまだ考慮しません。
